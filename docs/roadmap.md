# Hackagotchi — Product & Dev Roadmap (research dossier)

> **What this is.** Hackagotchi — the pocket UART debug bridge / black-box recorder in `hackagotchi/`
> (formerly "XIAO UART dev") — started as a refresh-immune telemetry tap for the PicoInky e-paper
> dashboard. This dossier captures the research behind turning it into a standalone, sellable
> instrument and a force-multiplier for our own dev loop. The product name is **Hackagotchi**; "XIAO
> RP2040" remains only as the hardware platform it runs on.
>
> **How it was produced.** A 5-lens multi-agent exploration (developer-experience, firmware
> features, hardware platform, market/productization, and an adversarial feasibility critic),
> pooled into 50 candidate ideas, run through a completeness-and-reality critic, then synthesized
> into the ranked roadmap below. Generated 2026-06-19.
>
> **Companion memory:** `xiao-multitool-product-roadmap.md`. **The tool itself:** see
> `hackagotchi/README.md` and the `xiao-uart-debug-bridge` memory.

## Part 1 — Synthesized roadmap (the recommendation)

*This is the decisive, de-duplicated plan. Parts 2 and 3 are the raw research it was distilled
from — kept so the reasoning behind each call is auditable.*


### PicoTap Roadmap — supercharge + sell

#### 1. Product thesis

**This is a flight recorder for dev boards whose own debug channel dies under load.** It taps a target's UART with a *separate* MCU + USB-CDC + hardware FIFO, so it keeps watching (and can drive) a board exactly when that board's own USB has wedged — during an e-paper refresh, a TLS handshake, or a crash — and it logs autonomously to its own SD card with an RTC wall-clock, on a screen, no host attached.

**The wedge, honestly:** narrow but real and unoccupied. Bus Pirate/Tigard/FT232H are tethered dumb adapters — kill the host, capture stops. Saleae needs a babysitting laptop. *Nothing* in the comparable set is a standalone, RTC-timestamped, log-on-boot field recorder with a screen and a reverse-drive channel. Do **not** sell it as a cheaper Bus Pirate or a scope/logic-analyzer — you lose that benchmark instantly. The moat is firmware behavior + docs, not silicon.

#### 2. Now / supercharge our dev (S-effort, do first)

- **Lenient marker decoder (the shared core).** One small host module: regex line → semantic field, with `uc>`/`uc<` pairing, `free=K` trend, `BUSY-TIMEOUT`/`FATAL` flags. Every other dev idea consumes this. Build once.
- **One-command flash-and-watch.** Wrapper: `deploy_all.sh` → inject `R` over the tap's UART RX → immediately tail the decoded XIAO stream. *This is the single highest-leverage dev win* — it gives boot/refresh visibility on a board whose USB goes dark exactly then. Best-effort `R`, fall back to BOOTSEL prompt if fully hung.
- **Black-box post-mortem decoder.** `log_NNN.txt` → timeline: last N markers, unmatched `uc>` (died mid-refresh), free-RAM floor, RTC wall-clock. Pure host tooling on data you already capture. "Where did it die and why" in one command.
- **Target-side last-gasp emit (`taplog.py`).** A tiny drop-in for the *target's* `except` handler that dumps traceback + free-RAM + page index out UART0 *before* it dies — the window where its own USB is already dead. Turns the black box from "the silence before" into "the cause." Generalizes as "include `taplog.py` in your project."
- **RTC + ms-delta timestamps in the black box.** Stamp marker bursts with wall-clock on idle-gap (frame boundary); keep relative-ms as always-present primary so a dead coin cell never loses ordering. Discipline `ticks_ms` against the per-flush RTC read so multi-day captures don't smear (needed for "wedge always ~03:00").
- **Stream substring-matcher → sniffer trigger.** Match on the raw incoming byte stream (not wrapped display lines): freeze/beep/LED/log-stamp on `ERROR`/`PANIC`/boot banner. This is also the shared primitive under assertions and macro `expect` — build it once, reuse three times.

#### 3. Next / firmware features that differentiate (M, ranked)

1. **Marker-map file format (first-class artifact).** A declarative `.json` per target: regex→field, thresholds, pairing rules. The on-device HUD, host decoder, and assertions all read *one* source. This is what turns "PicoInky-only" into "drop in a map for your board" — the real product surface the next three features circle.
2. **Soak watchdog / assertions.** Free-RAM floor, max refresh time, heartbeat-gap (= wedge detector), zero `FATAL`. On violation: beep piezo, flash OLED, stamp SD with FAIL. Guards the project's #1 historical bug class (ENOMEM). Thresholds live in the marker map so they version with the build.
3. **Live decoded health HUD (11th screen).** Page index, free-RAM bar, last refresh Δ, red flash on `BUSY-TIMEOUT`. A glanceable, refresh-immune instrument with no laptop — the thing a dumb dongle can never show. Config-driven via the marker map.
4. **REPL passthrough / escape mode.** Host escape (`{"cfg":{"mode":"repl"}}`) detaches the JSON sniff + line-assembly so you talk to the target's live REPL while its own USB is dead. Turns the tap from "I can see logs" into "I can poke the board." Pure software; low-collision escape so target traffic can't trip it.
5. **SD-image firmware self-update.** Drop `main.py` on the card, hold button at boot to copy-and-replace. The SD reader already exists — this defuses the single largest support cost (firmware updates for non-technical buyers) for almost free.
6. **Host-side macro scripts (send/wait/expect).** Scripts live on host, device keeps its 6-slot manual subset. Reuses shipped macros + the substring-matcher. Gives scripted UART conformance tests against *any* target. Resist a Turing-complete on-device DSL.
7. **Autobaud (best-guess).** Button on Baud screen measures shortest RX pulse via `time_pulse_us`, snaps to nearest rate. Frame as "best guess, confirm visually," not magic.

#### 4. Hardware path

**Stay on RP2040/RP2350. Do NOT go ESP32 as the main line, and do NOT add a radio.** No-WiFi/BT is a *feature*: it dodges FCC/CE intentional-radiator cert entirely, and a radio injects EMI into the exact analog/UART measurements you sell. Treat "wireless telemetry" as a trap, not a roadmap item.

- **Phase 1 (cheap, reversible): swap to XIAO RP2350.** Pin-for-pin, same expansion board, 2× SRAM (relieves scope/sniffer/SD buffers), more PIO. Budget it as **M not S** — "MicroPython runs as-is" is optimistic; you must re-validate SoftI2C OLED + sdcard.py + the bit-banged logic probe against RP2350 errata (GPIO pull-down latch E9 hits exactly those lines) on silicon with no test suite. The module swap is trivial; the bring-up is not.
- **Phase 1b: breadboard the protection front-end.** This is the **#1 must-do before selling**: today bare GPIO ties UART/ADC/PWM directly out. A 5V Arduino TX or reversed probe silently kills the unit. Add ~330R series + BAT54S clamps + a switchable level translator (validate on a scope at 115200 before committing — TXB/TXS auto-direction parts are picky on rise time).
- **Phase 2 (only after the feature set + pin map are FROZEN): one custom carrier PCB.** Socket the XIAO; bake in the protected/labeled probe header, fix the documented SD-vs-PWM pin collision in copper, and add LiPo charger + MAX17048 gauge (on the existing I2C bus). The LiPo completes the untethered black-box story you've 80% built. Note: LiPo triggers UN38.3 shipping rules — accept that cost knowingly.
- **Phase 3: labeled probe harness + fork the existing OpenSCAD case.** Ground clip + color-coded TX/RX/GND/SCOPE leads is what people actually buy a Tigard for; cheap, high perceived value. Cased unit reads as a product, not a prototype.
- **Accessories that matter:** the probe harness (high), the case (medium credibility), the LiPo+gauge (completes the pitch). **Skip:** INA219 power monitor — it collides at 0x40 with the AHT20 entry on the same SoftI2C bus, and at ~10 Hz it *aliases* the sub-second e-paper refresh spike it claims to measure, so it can't deliver its headline value.
- **Docs warning (genuine gap):** ground-loop hazard. Tapping a target on a different/mains-referenced supply ties your USB-host ground to theirs — protection resistors don't fix this. Ship a clear warning now; consider an isolated-UART SKU only if "tap anyone's board" demand proves real. Keep mono OLED default; color IPS is a far-future SKU (forces a 1700-line UI rewrite + worse battery).

#### 5. Productization

- **License first — this is blocking.** The repo is PolyForm Noncommercial; you cannot sell firmware that *is* the value under it, nor invite contributions. Relicense the XIAO subtree now while you're sole copyright holder (Apache-2.0 for adoption, or dual open-source + commercial). Do this before anything else ships.
- **What to sell:** firmware-first / SD-image-first. It's one self-contained `main.py` + `ssd1306.py` + `sdcard.py`, no cloud. Package a flashable bundle + pre-loaded SD image. Thousands of XIAO+expansion kits are already in drawers — you monetize an install base you didn't build, with near-zero COGS and zero fulfillment.
- **Two SKUs, not three. Drop the kit** — Seeed already sells the XIAO+expansion as a kit at a price you can't beat; you'd just resell theirs with an insert. Offer (a) firmware/SD-image download, (b) assembled, pre-flashed unit with SD + case where the margin lives.
- **OSHWA the design.** It's two off-the-shelf modules — nothing to protect, and OSH is a credibility/distribution multiplier in this exact crowd (Flipper/Bus Pirate/Glasgow all lean open). Monetize convenience, brand, support — not a hardware secret you don't have.
- **Pricing:** firmware $5–15 (Gumroad/Lemonsqueezy, name-your-price; it's a funnel, not a business). Assembled $45–59 (defensible: Bus Pirate 5 ~$30 bare, Saleae is $100s; the black-box framing justifies the premium). BOM ~$20–28 landed.
- **Channel:** own digital store for firmware now; Tindie for hand-assembled small batches (audience = exactly your buyers). Reserve Crowd Supply for a later integrated v2 with real volume ambition — premature for a two-module reflash.
- **Positioning + naming:** lead with the war story (a board whose hang killed its own only debug window — every embedded dev has lived it), not a spec sheet. Drop "XIAO" from the product name (it's a supported board, not the brand). Commit to a recorder identity (e.g. "BlackBox"-style + a distinctive trademark-able word, not "Black Box" alone). Generalize the reverse channel framing to "send bytes to any wedged target's RX" so it's not seen as a PicoInky one-off.
- **Docs are the moat.** Silicon is clonable; onboarding decides who wins (Bus Pirate's longevity is 80% docs). Ship a 5-min flash+first-capture quickstart, a wiring photo, copy-paste recipes ("record a hung board overnight", "sniff a sensor's UART"), and a troubleshooting page. Split a clean product manual out of the project-internal README.
- **MVP-for-sale:** relicensed firmware + the SD black-box recorder + decoded post-mortem tool + REPL passthrough + SD-image self-update + the protection front-end validated + a real quickstart/manual. Hardware harden the boot/SD/crash paths (the crash handler currently prints and spins forever — looks bricked; add watchdog/auto-reboot, visible "logging stopped" state, SD-full handling, documented recovery flash). Those unglamorous fixes set your return rate.

#### 6. Don't bother / traps

- **Glitch / fault injection — dead. Don't revive.** RP2040 at 125 MHz via MicroPython PIO is far too jittery; no glitch hardware on the board. It would badly underdeliver vs the name and invite "why isn't it ChipWhisperer" support load.
- **Don't market the scope / logic analyzer / PWM as instruments.** Screen 3 is one ADC pin busy-polled in a Python loop (timebase is fiction above a few hundred Hz); screen 4 samples once per ~80 ms render (~12 Hz); PWM meter floors ~100 Hz. A reviewer with a Saleae benchmarks these and the *whole product* loses credibility. Keep them as honest bonuses ("a quick pocket probe when you need one"), never headlines.
- **Don't chase Bus Pirate/Flipper/Tigard protocol parity.** You have a hardcoded-name-table I2C scanner, no SPI/JTAG/SWD, no level shifting yet. "Number of protocols" is a losing race against years of head start. Every "just add SPI/JTAG too" request is scope creep toward a fight you lose.
- **Don't grow the on-device macro store or build a Turing-complete on-device DSL.** 6×14-char slots, one button, 6-line OLED. Real scripts live host-side; device macros stay a manual subset.
- **Don't add more physical buttons.** That means a custom board = BOM/regulatory/scope detonation. Solve the single-button UX ceiling host-side (the JSON + single-char channel already exists); make the screen glanceable/read-only and let a thin companion CLI navigate. Keep that CLI a thin documented protocol + reference tool — not an Electron app.
- **Don't spin the carrier PCB until the pin map is frozen.** Premature board spin is the classic hardware money-sink; every screen that changes pin usage forces a respin.
- **Don't add a radio or a LiPo casually.** Wireless retroactively pulls the whole SKU into FCC/CE cert; LiPo into UN38.3. Stay wired + coin-cell for the cheapest regulatory path; add LiPo only on the carrier, knowingly.
- **PIO protocol decoders: ship exactly ONE, and only after RP2350 + carrier.** Don't list "five decoders" — it's weeks of hardware-in-the-loop with no test suite, and it's *gated on* the hardware path (free pins + state machines), not a standalone item. Async-UART-framing or 1-Wire first; multi-master I2C is hardest, last.

## Part 3 — Completeness & reality critic

*An adversarial pass over the full 50-idea pool: what was missing, what was wrong or double-counted, and the single highest-leverage dev win + strongest commercial wedge.*

I have enough to work with from the pool and the project context. Let me deliver the critic analysis directly.

---

### (1) HIGH-VALUE IDEAS MISSING FROM THE POOL

**A. A target-agnostic "marker map" file format + a small registry of community maps.** Three separate ideas (live decode HUD, soak assertions, crash decoder) each independently note they depend on a "per-target marker map" and each waves at it as a footnote/risk. Nobody proposes the marker map *itself* as a first-class deliverable: a small declarative schema (regex → semantic field, plus thresholds and pairing rules like `uc>`/`uc<`) that the firmware HUD, the host decoder, AND the assertions all consume from one source. This is the actual product surface the three dev-leverage ideas keep circling — it should be one named artifact, not three implicit copies. It also turns "works for PicoInky only" into "drop in a .json for your board."

**B. Crash-handler telemetry emission on the *target* side (close the loop the other way).** The whole pool treats the XIAO as a passive recorder of whatever the target happens to print. Nobody proposes the obvious complement: a tiny drop-in MicroPython snippet/decorator for the *target* firmware that, in its `except`/last-gasp handler, dumps a final structured frame (traceback, free-RAM, page index) out UART0 *before* it dies — precisely the window where its own USB-CDC is already dead. The XIAO's value is "I see the board after USB wedges"; pairing it with a guaranteed last-gasp emit on the target makes the black box capture the *cause*, not just the silence before it. Low effort, and it generalizes as "include `taplog.py` in your project."

**C. Galvanic isolation / common-ground hazard.** The hardware lens covers level shifting and over-voltage clamping thoroughly but never mentions ground-loop / floating-ground risk: tapping a UART on a target powered from a different supply (or a mains-referenced board) ties the XIAO's USB-host ground to the target ground. For a "clip it onto any flaky board" pitch this is a real fry/safety path that protection resistors don't address. At minimum it's a docs warning; at most an isolated-UART SKU. It's a genuine gap given the "tap anyone's board" ambition.

**D. Firmware self-update / recovery path as a shipped feature.** The feasibility lens flags "firmware updates over USB to non-technical users" as a *support burden* but nobody proposes the mitigation: since the device already has an SD card, an SD slot can *be* the update channel (drop `main.py` on the card, hold button at boot to copy-and-replace), sidestepping mpremote entirely for buyers. This directly defuses the single largest identified support cost and is uniquely cheap here because the SD reader already exists.

**E. Timestamp-source clock-drift / discipline.** Multiple ideas lean on RTC + `ticks_ms` for the black-box timeline, but nobody notes that `ticks_ms` drifts vs the PCF8563 over a multi-day capture, nor proposes disciplining the ms counter against the once-per-flush RTC read. For "wedge always ~03:00" field correlation (idea explicitly claims this) the two clocks must be reconciled or long captures smear. Small but load-bearing for the exact use the RTC ideas sell.

### (2) WRONG / DOUBLE-COUNTED / MIS-ESTIMATED

**Heavy double-counting in the SD-timestamp + replay cluster.** Four ideas are substantially the same two features:
- "Captured-frame timestamps in the black box" (fw-features) and "RTC-correlated wall-clock event log" (dev-leverage) are the **same feature** — per-frame/per-batch timestamping of the SD log using RTC + ms-delta. They differ only in framing (idle-gap frame boundary vs per-flush stamp). Count as one S-effort item.
- "Save-capture-to-SD with replay/inject" (fw-features) and "Captured-session replay into the bridge" (dev-leverage) are the **same replay feature** from two ends. The fw one even says it depends on the timestamp feature; the dev one is the host-side reader of the same artifact. One feature, two interfaces.

**"Macros-as-test-scripts" (dev, S) vs "Scriptable macro language" (fw, M, overhyped).** These overlap heavily — both are "turn 6 fixed macros into send/wait/expect sequences." The dev lens correctly resolves the tension the fw lens flags: scripts live host-side, device macros stay a 6-slot manual subset. So the fw "M + overhyped" estimate is right to be cautious, but the two entries should be merged: host-side scripting = S; on-device DSL = correctly rejected. Don't bank both.

**"Trigger/filter/search on sniffer" is a hidden dependency, not a standalone S.** Two other ideas ("scriptable macro language" expect, "soak assertions") quietly assume substring-match-on-stream exists. It's fine as S, but it's a **prerequisite primitive**, so its true value is as the shared substring-matcher behind assertions/expect/trigger — building it three times would be the mis-estimate.

**RP2350 swap "effort: S" is optimistic-but-defensibly-flagged.** The entry itself caveats that the PIO logic-analyzer work is the "L part, not the swap," which is honest. But two risks are under-weighted: (a) MicroPython on RP2350 + the **SoftI2C OLED + sdcard.py + single-file 1730-line firmware** is not guaranteed drop-in — RP2350 errata E9 (the GPIO input pull-down latch) can affect exactly the bit-banged logic-probe / SoftI2C lines this firmware leans on, and (b) "MicroPython runs as-is" assumes the SD/SoftI2C drivers behave identically. The *module swap* is S; "and everything works unchanged" is closer to M once you account for re-validating every screen on new silicon with no test suite. Not wrong, but the headline S undersells bring-up.

**"INA219 at 0x40" — the collision claim is correct and is the strongest factual catch in the pool** (0x40 vs the AHT20 entry in KNOWN_I2C, on the *same* SoftI2C bus as OLED/RTC). Worth elevating, not flagging. One refinement: it claims "1–10 Hz power logging, not transients" — correct, but that means it **cannot** see the e-paper-refresh current spike it promises to correlate ("what the refresh costs in mA"). The refresh transient is sub-second; SoftI2C INA219 at ~10 Hz will alias it. The *value* sentence over-claims relative to its own *risk* sentence. Mis-estimated value, not effort.

**Glitch/fault injection (overhyped: true) — correctly killed, and the reasoning is sound.** No correction needed; the "RP2040 PIO is too jittery, no glitch hardware" verdict is right. Keep it dead.

**PIO protocol decoders (L, overhyped) — the "ship ONE" framing is correct, but the pin-budget claim slightly overstates.** It says SD owns SPI0, I2C owns OLED/RTC, UART owns D6/D7 leaving "few free Grove pins." True today, but this is the one feature that the RP2350 + carrier-PCB ideas exist to unblock (PIO state machines + freed pins). So its real effort is **gated on the hardware path**, not standalone L — listing it as an independent L item double-counts the RP2350/carrier investment that the hw lens already budgets.

**No factually-wrong physics elsewhere.** The scope/PWM/throughput honesty entries (per-char TX ceiling, ADC-loop fictional timebase, 12 Hz logic probe) are all technically accurate and align with the firmware description; the feasibility lens is the most reality-anchored set.

### (3) THE TWO HIGHEST-LEVERAGE PICKS

**Highest-leverage for OUR OWN dev workflow: "One-command flash-and-watch" (deploy → inject `R` over the tap → tail decoded boot).**
This is the single idea that attacks the project's #1 documented daily pain — the Pico W's USB-CDC goes dark *exactly at boot and during refresh/TLS*, which is the one moment you most need to see. The tap is already the only live window then; nobody has wired deploy→reboot→observe into one loop. It's S effort, pure host tooling, depends only on already-shipped primitives (deploy scripts + the `R` control char + a lenient marker tail). It compounds: every other dev-leverage idea (decoder, assertions, bisect) becomes useful *through* this loop. The caveat in its own risk note (`R` only works if the target can still drain RX; else BOOTSEL) is the correct, honest bound. Build the lenient marker decoder as the shared core under it (that's missing-idea A), and you've simultaneously seeded the HUD, soak watchdog, and crash post-mortem.

**Strongest commercial wedge: "Black box for wedged dev boards" — the autonomous, screen-on, SD-logging, refresh/USB-immune recorder, sold firmware-first.**
The pool reaches near-unanimous convergence here across three independent lenses (market, feasibility, dev-leverage all land on the same one-liner), which is itself signal. It is the *only* defensible niche against the comparable set: Bus Pirate/Tigard/FT232H die when the host dies; Saleae needs a babysitting laptop; none record standalone to SD with a wall clock and a screen. The wedge is firmware behavior (log_on_boot + 60s heartbeat bounding time-of-death + RTC session header + reverse-drive channel), not silicon — so the go-to-market is **firmware-first / SD-image-first** onto the thousands of XIAO+expansion kits already in drawers: near-zero COGS, no fulfillment, instant distribution, and it doubles as the funnel for a later assembled SKU. Two hard gates the pool correctly flags must be cleared *first*: (i) **fix the PolyForm-Noncommercial license** — it currently forbids the exact commercial use being planned and is a true blocker, and (ii) **do not headline the scope/logic-analyzer/PWM** — technically-literate buyers will benchmark them and discount the whole product; keep them as bonuses. The moat is the recorder UX + docs, and the war story (a board whose hang killed its own only debug window) is the marketing that no spec sheet can copy.
